## Entity Definitions

### 1. User Profile

**Purpose**: Represents application users with their roles and attributes. Linked to Supabase authentication system.

**Attributes**:
- `id` UUID PRIMARY KEY
  - Generated by `gen_random_uuid()`
  - Unique identifier for the user profile
  
- `auth_user_id` UUID NOT NULL
  - Foreign key to auth.users(id)
  - ON DELETE CASCADE (user profile deleted when auth user is deleted)
  - Used to link profile to authentication system
  
- `role` role_enum NOT NULL
  - Enum: super_admin, content_manager, teacher, viewer, student
  - Determines access permissions across the application
  - Set during user creation or updated by super_admin
  
- `display_name` TEXT NOT NULL
  - User's display name in the interface
  - Can be different from email address
  - Required for all users
  
- `avatar_url` TEXT
  - URL to user's profile image
  - Optional field
  - Defaults to null
  
- `grade_level` TEXT
  - Academic grade level (e.g., "Grade 5", "Year 10")
  - Optional field
  - Used for filtering and reporting
  
- `is_active` BOOLEAN DEFAULT true
  - Whether the user account is active
  - Inactive users cannot log in
  - Can be toggled by super_admin
  
- `created_at` TIMESTAMPTZ DEFAULT NOW()
  - When the profile was created
  - Automatically set
  
- `updated_at` TIMESTAMPTZ
  - When the profile was last updated
  - Updated via trigger
  
- `deleted_at` TIMESTAMPTZ
  - Soft delete timestamp
  - When user was deactivated (not deleted)
  - Allows for recovery of user data

**Relationships**:
- One-to-one with auth.users table
- One-to-many with quiz_attempts (user takes quizzes)
- One-to-many with lesson_completions (user completes lessons)
- One-to-many with point_transactions (user earns/spends points)

### 2. User Session

**Purpose**: Represents authentication session state for logged-in users.

**Attributes**:
- `access_token` TEXT NOT NULL
  - JWT token from Supabase Auth
  - Contains user claims including role
  - Used for API requests and authorization
  
- `refresh_token` TEXT
  - Token for refreshing the session
  - Handled by Supabase Auth helpers
  - Not stored directly in application
  
- `expires_at` TIMESTAMPTZ NOT NULL
  - When the access token expires
  - Used for session validation
  - Typically 1 hour from creation
  
- `user_role` role_enum NOT NULL
  - User's role extracted from JWT claims
  - Cached for quick authorization
  - Matches role in user_profile
  
- `created_at` TIMESTAMPTZ DEFAULT NOW()
  - When the session was created
  - Used for session timeout calculation

**Relationships**:
- Associated with one user profile via JWT claims
- Created and managed by authentication middleware

### 3. Environment Configuration

**Purpose**: Stores environment variables and configuration needed for the application to run.

**Attributes**:
- `key` TEXT PRIMARY KEY
  - Environment variable name
  - e.g., "SUPABASE_URL", "NEXTAUTH_URL"
  
- `value` TEXT NOT NULL
  - Environment variable value
  - May contain secrets (handled carefully)
  
- `is_sensitive` BOOLEAN DEFAULT false
  - Whether the value contains sensitive data
  - Controls logging and display behavior
  
- `description` TEXT
  - Purpose of the configuration
  - Helps developers understand usage
  
- `last_updated` TIMESTAMPTZ DEFAULT NOW()
  - When the value was last changed
  - For audit trail

**Relationships**:
- Used by all components that need environment configuration
- Updated by deployment scripts or manual configuration

## Validation Rules

### User Profile Validation
- `display_name`: Must not be empty, max 100 characters
- `grade_level`: Must match pattern if provided (e.g., "Grade \d+", "Year \d+")
- `role`: Must be one of the defined enum values
- `auth_user_id`: Must be a valid UUID and exist in auth.users table

### Session Validation
- `access_token`: Must be a valid JWT with required claims
- `expires_at`: Must be in the future
- `user_role`: Must match role in user_profile

### Environment Configuration
- `key`: Must follow naming convention (UPPER_SNAKE_CASE)
- `value`: Required, max 2048 characters
- `is_sensitive`: Must be boolean

## State Transitions

### User Profile States
```text
Active → Inactive
  Trigger: super_admin sets is_active = false
  Effect: User cannot log in, but data preserved
  
Inactive → Active
  Trigger: super_admin sets is_active = true
  Effect: User can log in again
```

### Session States
```text
Valid → Expired
  Trigger: expires_at < NOW()
  Effect: User must re-authenticate
  
Valid → Revoked
  Trigger: User logs out or password changed
  Effect: Session invalidated immediately
```

## Data Access Patterns

### Read Operations
1. **User by ID**: `SELECT * FROM user_profiles WHERE id = $1 AND deleted_at IS NULL`
2. **User by Auth ID**: `SELECT * FROM user_profiles WHERE auth_user_id = $1 AND deleted_at IS NULL`
3. **Active Users**: `SELECT * FROM user_profiles WHERE is_active = true AND deleted_at IS NULL`
4. **Users by Role**: `SELECT * FROM user_profiles WHERE role = $1 AND is_active = true AND deleted_at IS NULL`

### Write Operations
1. **Create User**: `INSERT INTO user_profiles (...) VALUES (...)`
2. **Update User**: `UPDATE user_profiles SET ... WHERE id = $1`
3. **Soft Delete**: `UPDATE user_profiles SET deleted_at = NOW() WHERE id = $1`

### Indexes (for performance)
```sql
CREATE INDEX idx_user_profiles_auth_user ON user_profiles(auth_user_id);
CREATE INDEX idx_user_profiles_role ON user_profiles(role);
CREATE INDEX idx_user_profiles_active ON user_profiles(is_active, created_at DESC);
```

## Security Considerations

### Row Level Security (RLS)
- All tables have RLS enabled with default deny
- Users can only see their own profiles unless they have higher permissions
- super_admin can see all active user profiles

### Data Protection
- Sensitive environment variables are not logged
- User passwords are never stored directly (handled by Supabase Auth)
- All database access goes through Supabase client helpers

### Audit Trail
- User profile changes are logged in activity_logs
- Session creation/termination is logged
- Environment variable changes are tracked
